## 一、项目目标（明确、可答辩）

### 总目标
设计并实现一个：

> **Android 运行环境可信度评估系统**

用于在 **App 侧 + Native 层** 对运行环境进行评估，识别：
- Hook / 注入（Frida / Xposed / Zygisk）
- 改机 / 定制 ROM
- 模拟器 / 虚拟化环境

最终输出一个：
> **风险评分（Risk Score） + 可信等级判断**

而不是“单点 yes / no 检测”。

## 二、系统整体架构（答辩重点）

```
┌────────────┐
│ Java Layer │  ← 行为 / Framework / Runtime
└─────┬──────┘
      │ JNI
┌─────▼──────┐
│ Native     │  ← ELF / linker / maps / syscall
└─────┬──────┘
      │ 结果汇总
┌─────▼──────┐
│ Risk Core  │  ← 风险聚合 / 权重 / 策略
└─────┬──────┘
      │ 输出
┌─────▼──────┐
│ Decision   │  ← 可信等级
└────────────┘
```

## 三、功能模块拆解（必须完成）

### Module 1：Hook / 注入检测模块（Native 核心）

#### 检测维度（至少 3 类）
- `/proc/self/maps` 异常
- linker so 加载异常
- inline hook 指令特征

#### 要求
- **不能只检测 Frida 字符串**
- 至少一个 **行为级 / 结构级检测**

---

### Module 2：ART / Runtime 异常检测（Java + Native）

#### 示例方向
- Method 入口异常
- call stack 非预期路径
- oat / dex 加载差异

---

### Module 3：系统改机与 ROM 特征分析

#### 检测方向
- build.prop 一致性
- selinux 状态
- 系统服务差异

#### 重点
- **解释为什么 LineageOS 不等于官方 ROM**

---

### Module 4：虚拟化 / 模拟环境识别

#### 维度
- sensor 行为
- timing / syscall 延迟
- binder 行为

---

### Module 5：风险评分系统（核心亮点）

#### 设计要求
- 每个检测点 → 子风险值
- 权重可配置
- 输出：
  - Low / Medium / High Risk

#### 必须说明
- 为什么不用单点判断
- 如何避免误杀
